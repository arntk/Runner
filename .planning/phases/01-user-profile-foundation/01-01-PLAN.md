# Plan 01-01: Node.js Auth & Profile Backend (Wave 1)

**Goal**: Establish the primary API gateway for user management, authentication, and profile persistence.

## Wave 0: Infrastructure & Testing

### Task 01-01-00: Initialize Test Stubs
- <read_first> `.planning/phases/01-user-profile-foundation/01-VALIDATION.md`
- <action> Create `backend/tests/auth.test.js` and `backend/tests/profile.test.js` with failing test cases (stubs) for registration, login, and profile CRUD as defined in the Validation Strategy.
- <acceptance_criteria> `npm test` runs and reports these tests as failed/pending.

## Infrastructure Setup

### Task 01-01-01: Initialize Backend Service
- <read_first> `docker-compose.yml`
- <action> Add `backend` service to `docker-compose.yml` using `node:20-alpine`, mapping host port `3001` to container port `3001`. Define environment variables: `DATABASE_URL`, `JWT_SECRET`. Create a minimal `backend/package.json` with `express` and `jest` dependencies.
- <acceptance_criteria> `docker-compose up backend` starts without errors; `grep -q "backend:" docker-compose.yml` returns true.

### Task 01-01-02: Setup Database Schema
- <read_first> `analytics_engine/models.py`
- <action> Create `backend/schema.sql`. Define `users` table (id UUID, email UNIQUE, password_hash, created_at) and `profiles` table (user_id FK, name, max_hr, resting_hr, hr_zones JSON, pace_zones JSON, vdot FLOAT, hr_unit_pref VARCHAR(10) DEFAULT 'absolute', pace_unit_pref VARCHAR(10) DEFAULT 'absolute'). The `hr_zones` and `pace_zones` JSON must support storing both value and unit type per zone (e.g., `{"min": 140, "max": 150, "unit": "bpm"}` or `{"min": 80, "max": 85, "unit": "percent_max_hr"}`).
- <acceptance_criteria> `psql -d runners -f backend/schema.sql` executes successfully; `\dt` shows `users` and `profiles` tables.

## Authentication API

### Task 01-01-03: Implement Registration & Login
- <read_first> `backend/server.js`
- <action> Install `bcryptjs`, `jsonwebtoken`, `pg`. Implement `POST /api/auth/register` (hash password, insert into `users`) and `POST /api/auth/login` (verify hash, return JWT). Ensure registration also creates a default entry in the `profiles` table.
- <acceptance_criteria> `curl -X POST -H "Content-Type: application/json" -d '{"email":"test@example.com","password":"password123"}' http://localhost:3001/api/auth/register` returns 201.

### Task 01-01-04: Auth Middleware
- <read_first> `backend/server.js`
- <action> Implement `authenticateJWT` middleware that extracts the `user_id` from the `Authorization: Bearer <token>` header, verifies the signature, and attaches the decoded `user_id` to `req.user`.
- <acceptance_criteria> Protected routes return 401 without a valid token and 200 with one.

## Profile Management

### Task 01-01-05: Profile CRUD Endpoints
- <read_first> `.planning/phases/01-user-profile-foundation/01-CONTEXT.md` (Decision D-08)
- <action> Implement `GET /api/profile` and `PUT /api/profile`. The `PUT` endpoint must accept and persist `hr_zones` and `pace_zones` as JSON, supporting both absolute values (BPM, min/km) and relative percentages (% of Max HR, % of Threshold Pace) for each of the 5 zones (Recovery, Aerobic Base, Grey Zone, Threshold, VO2 Max).
- <acceptance_criteria> `PUT /api/profile` with JSON body containing relative units updates the database; `GET /api/profile` returns the same units.

- <automated> `npm test backend/tests/profile.test.js`
